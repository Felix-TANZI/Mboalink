Resume de notre travail sur MboaLink
Date de creation : 29/06/2026

1. Objectif general

Nous avons travaille sur l'application MboaLink afin de clarifier les roles, stabiliser le backend/front, preparer l'integration FreeRADIUS, ameliorer l'interface administrateur et documenter l'etat du projet.

L'objectif principal actuel est d'avoir une application utilisable par l'equipe MboaLink et suffisamment stable pour permettre ensuite a l'equipe reseau/FreeRADIUS de l'utiliser et de tester l'integration avec les equipements reels.


2. Roles et logique d'acces

Les roles principaux retenus sont :

- ADMIN : super administrateur MboaLink. Il dispose maintenant d'une interface dediee, separee de l'ancienne interface operationnelle.
- SUPPORT : support IT MboaLink. Il garde l'ancienne interface operationnelle, mais sans Hotel Manager et sans Device Manager.
- HOTEL_IT : informaticien de l'hotel. Il gere les vues utiles a son hotel : dashboard hotel, WiFi Code, statut des logins, rooms, config WiFi, devices, portail captif/web site manager selon les droits prevus.
- RECEPTIONIST : receptionniste de l'hotel. Il est limite a la vue Manual Login.
- CLIENT : role existant, mais pas encore une vraie vue admin dediee.

La logique decidee :

- ADMIN ne doit plus utiliser l'ancienne interface bleue.
- ADMIN doit arriver sur la nouvelle interface /admin-mboa.
- SUPPORT conserve l'ancienne interface mais sans gestion hotel et sans gestion appareils/equipements.
- HOTEL_IT et RECEPTIONIST sont automatiquement limites a leur hotel.
- ADMIN et SUPPORT peuvent voir des donnees globales ou filtrees selon les modules autorises.

11. Probleme PostgreSQL backend MboaLink

Probleme rencontre :

- Le backend crashait avec Prisma P1000.
- PostgreSQL avait un volume deja initialise avec un autre mot de passe.
- Le backend essayait de se connecter avec postgres/postgres.

Commande corrective utilisee :

docker exec -it core_access_and_identity_service_back-postgres-1 \
  psql -U postgres -d mboalink_local \
  -c "ALTER USER postgres WITH PASSWORD 'postgres';"

Ameliorations faites :

- docker-compose backend stabilise.
- Ajout des variables POSTGRES_USER, POSTGRES_PASSWORD, POSTGRES_DB.
- Ajout d'un script d'init PostgreSQL backend :
  postgres-init/01-enforce-postgres-password.sql

Ce script contient :

ALTER USER postgres WITH PASSWORD 'postgres';

Remarque :

Ce script s'applique aux nouveaux volumes. Pour un volume deja initialise, il faut faire une derniere correction manuelle ou recreer le volume.


12. Portail captif et Radius

Le portail captif a ete teste.

Les tests backend lies au portail captif ont valide :

- Connexion par UUID.
- Connexion par code WiFi.
- Connexion par nom + chambre.
- Rejet des passes expires/revoques/utilises.

Le portail captif tourne dans le dossier radius.

Cas conferences / evenements :

Les personnes qui recoivent un code lors d'une conference devront aussi passer par le portail captif pour acceder au Wi-Fi. La difference est qu'elles ne se connecteront pas avec nom + chambre, mais avec un code evenement/conference, assimilable a un voucher ou a un guest pass collectif.

Logique recommandee :

- Le portail captif reste le point d'entree unique pour les clients, invites et participants.
- Un participant a une conference saisit simplement son code conference sur le portail captif.
- Il n'a pas besoin d'un compte dans l'administration MboaLink.
- Le code conference peut etre limite par date, duree, nombre d'utilisations, hotel/lieu, bande passante et zone Wi-Fi.
- Cote FreeRADIUS, ce code devra etre transforme en autorisation reseau comme les autres guest passes.

Cela permet de garder une seule logique d'acces Wi-Fi : tout passe par le portail captif, mais les modes d'identification changent selon le contexte.

Implementation ajoutee :

- Le champ principal du portail captif indique maintenant clairement : code Wi-Fi, code conference ou UUID.
- Le portail captif envoie ce champ comme `code` et comme `uuid` afin de couvrir les deux logiques backend deja existantes.
- Les guest pass simples et les mass guest pass restent donc utilisables directement depuis le portail captif.
- Les codes generes dans WiFi Code, y compris les codes de conference/evenement, doivent etre saisis par l'utilisateur sur le portail captif.

Decision supplementaire sur HOTEL_IT :

- HOTEL_IT ne doit plus voir Hotel Manager.
- HOTEL_IT ne doit plus voir Device Manager.
- La navigation HOTEL_IT est maintenant rapprochee de celle du SUPPORT : dashboard, logins utiles et notifications.
- Les routes directes rooms, config WiFi et devices ne sont plus accessibles a HOTEL_IT cote frontend.
- Les mutations backend hotel, rooms et config WiFi ont ete reservees a ADMIN.


13. Probleme FreeRADIUS / hilton_freeradius

Plusieurs problemes ont ete observes sur le conteneur hilton_freeradius :

- Conteneur unhealthy.
- Restart loop.
- docker logs parfois vide.
- PostgreSQL saturait en connexions : "sorry, too many clients already".
- FreeRADIUS avait un pool SQL trop grand.
- Une mauvaise commande sed avait introduit des caracteres \n litteraux dans sql.conf.
- Ensuite, FreeRADIUS echouait avec :
  password authentication failed for user postgres
- Puis PostgreSQL indiquait :
  role "postgres" is not permitted to log in

Corrections faites/proposees :

- Reduction du pool SQL FreeRADIUS :
  start = 1
  min = 1
  max = 3
  spare = 1
  retry_delay = 1
  max_retries = 3
  cleanup_interval = 30

- Suppression des anciennes options :
  num_sql_socks
  connect_failure_retry_delay

- Ajout dans radius/init.sql :

ALTER ROLE postgres WITH LOGIN PASSWORD 'postgres';

Important :

Cette ligne corrige les nouveaux volumes PostgreSQL Radius.
Pour appliquer sur le serveur actuel, il faut probablement recreer le volume Radius :

cd /home/junior/Mboalink/radius
docker compose down -v
docker compose up -d --build

Attention :

docker compose down -v supprime la base radius_hotel du module Radius. Le fichier init.sql recrée les tables, mais les donnees existantes sont perdues.

17. Decision de deploiement

Pour l'instant, il a ete retenu que MboaLink n'est pas encore suffisamment finalise pour passer a un deploiement professionnel complet.

Railway ou le deploiement actuel peut encore etre utilise pendant la finalisation.

Le passage vers un deploiement plus professionnel devra attendre :

- Stabilisation FreeRADIUS.
- Validation du portail captif avec equipements reels.
- Confirmation des roles avec les responsables MboaLink.
- Tests bout en bout frontend, backend, portail captif et reseau.
